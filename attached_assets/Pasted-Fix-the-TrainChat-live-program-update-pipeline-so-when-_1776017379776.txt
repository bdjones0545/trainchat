Fix the TrainChat live program update pipeline so when the user modifies the program (example: change from 3 days/week to 4 days/week), ALL dependent UI and state layers update consistently.

CURRENT ISSUE:
The AI successfully rebuilt the program as a 4-day plan, but several areas still show stale 3-day data.

Observed problems:
- Program summary/header still references “3-Day” in title or description
- Some descriptive text still says “A 3-day performance program...”
- Changes tab does not show the update
- History tab does not show a new version
- Some card metadata is stale even though the session structure changed
- The actual exercises/days were regenerated, but the surrounding system did not fully sync

GOAL:
Make the live program system fully state-driven so any accepted AI modification updates:
1. the underlying program
2. the visible summary/header copy
3. version history
4. changes log
5. all dependent views and cards

----------------------------------------
1. IDENTIFY THE ROOT ISSUE
----------------------------------------

Audit the update flow for program modifications.

Likely problem:
- the new program object is being partially replaced
- but derived display fields are cached, hardcoded, or not recomputed
- versioning/history is not triggered for refinement mutations
- summary cards are reading stale state instead of current programState

Find where:
- title
- frequency
- summary description
- badge data
- history entries
- changes entries

are generated and ensure they come from the latest programState, not initial build state.

----------------------------------------
2. CREATE A SINGLE SOURCE OF TRUTH
----------------------------------------

All live program UI must derive from one central object, for example:

programState = {
  version,
  currentProgram,
  currentSummary,
  changes,
  history,
  metadata,
  updatedAt
}

Where currentProgram includes:
- frequencyPerWeek
- title
- description
- sessions[]
- block info
- goal
- sport context

Do not allow tabs/components to independently hold stale copies of program metadata.

----------------------------------------
3. FORCE FULL RECOMPUTE AFTER PROGRAM REFINEMENT
----------------------------------------

When the user says something like:
- “Make it a 4 day program”
- “Shorten the sessions”
- “Swap deadlifts for trap bar”
- “Make this more power focused”

The refinement pipeline must:

1. interpret intent
2. update programState.currentProgram
3. regenerate summary metadata
4. increment version
5. create changes entry
6. create history snapshot
7. re-render all dependent tabs

This must happen every time a meaningful program mutation is accepted.

----------------------------------------
4. FIX PROGRAM HEADER / SUMMARY SYNC
----------------------------------------

After regeneration, update all visible summary fields including:
- title
- frequency
- description
- block summary
- session count
- badge labels

Example:

- “Hockey Performance Program — 4-Day”

- description text should no longer say “A 3-day performance program...”

Do not let the title and description lag behind the rebuilt sessions.

----------------------------------------
5. FIX CHANGES TAB
----------------------------------------

When a refinement occurs, automatically create a new changes entry.

Example for this case:

- Changed frequency from 3 days/week to 4 days/week
- Redistributed weekly volume across 4 sessions
- Preserved hockey performance goal and core movement intent
- Reason: user requested a 4-day structure

The Changes tab should never stay frozen at only V1 if the program was meaningfully modified.

Example:

- Athletic Performance
- 4d/Wk
- Hockey

Summary:

History should show:
- V1 initial build
- V2 refined build

The current live version badge should move to the newest version.

----------------------------------------
7. SNAPSHOT THE FULL PROGRAM ON EACH VERSION
----------------------------------------

For every meaningful update, store a full snapshot:

historyEntry = {
  version: 2,
  title: "Frequency Update",
  timestamp,
  tags: ["Athletic Performance", "4d/Wk", "Hockey"],
  summary: "...",
  snapshot: fullProgramObject
}

Do not store only metadata.
Store the full program so history is real and recoverable.

It should generate:
- displayTitle
- frequencyLabel
- summaryDescription
- sessionCount
- estimatedDuration
- goal tags
- sport tags

Every time currentProgram changes, rerun this function.

Do not manually patch strings in scattered components.
