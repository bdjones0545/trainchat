We are now implementing:

GLOBAL LEARNING LAYER (CONTROLLED PRODUCT-WIDE LEARNING)

OBJECTIVE:
Allow TrainChat to improve over time from aggregate user interactions across all users — without allowing the live agent to self-rewrite unsafely from raw conversations.

This is NOT a self-modifying AI brain.
This is a controlled global learning pipeline.

The goal is:
- capture structured signals from product usage
- identify repeated successful/unsuccessful patterns
- aggregate them across users
- surface high-confidence improvements
- promote those improvements intentionally into the core system

==================================================
CORE PRINCIPLE
==================================================

The live agent should NOT directly learn from every conversation in an uncontrolled way.

Instead:

1. User interactions generate structured learning signals
2. Signals are stored and aggregated
3. Patterns are scored for confidence
4. Candidate improvements are surfaced
5. Only reviewed/high-confidence patterns are promoted into:
   - routing rules
   - prompts
   - validators
   - transformation rules
   - exercise relationships
   - anti-pattern filters
   - example banks

Stable Core + Global Learning Feedback Loop

==================================================
TASK 3 — DEFINE WHAT THE SYSTEM SHOULD LEARN GLOBALLY
==================================================

The global layer should learn patterns like:

A. ROUTING IMPROVEMENTS
- which phrases should route deterministic vs agent
- where deterministic path fails too often
- where default execution should auto-apply

B. TRANSFORMATION IMPROVEMENTS
- what users usually mean by “make it harder”
- what “add more exercises” usually resolves to
- what kinds of transformations are accepted vs reverted

C. EXERCISE GRAPH IMPROVEMENTS
- which substitutions are accepted frequently
- which substitutions are often rejected or immediately re-edited
- which harder/easier mappings work best

D. CLARIFICATION REDUCTION
- which prompts trigger unnecessary clarification
- which assumptions were safe and well accepted

E. ADHERENCE / PRACTICALITY SIGNALS
- session length patterns
- completion vs dropout patterns
- where complexity is too high
- where users skip certain modalities or movement types

F. PROGRAM QUALITY SIGNALS
- outputs that frequently require a second edit
- outputs that cause immediate rollback or dissatisfaction
- validators that fire repeatedly on specific request classes

==================================================
TASK 4 — NORMALIZE LEARNING SIGNALS
==================================================

Before aggregation, normalize events into reusable learning keys.

Examples:
- “make this harder”
- “harder please”
- “too easy”
→ normalizedRequestKey = "increase_difficulty_general"

- “add more to day 1”
- “add more exercises to day 1”
- “expand day 1”
→ normalizedRequestKey = "expand_session_volume_day"

This prevents the learning layer from fragmenting around surface wording.

TASK 7 — CREATE A CONFIDENCE / PROMOTION MODEL
==================================================

Each candidate improvement should have:
- evidence count
- success/failure patterns
- confidence score
- safety sensitivity
- promotion recommendation

Examples:
- low-risk routing improvements can promote faster
- safety-sensitive clinical or injury-related patterns should require stricter review
- programming-rule changes require stronger evidence than UI routing changes

Add a model like:

type LearningCandidate = {
  id: string;
  type: LearningCandidateType;
  key: string;
  summary: string;
  evidenceCount: number;
  confidenceScore: number;
  riskLevel: "low" | "medium" | "high";
  recommendation: "review" | "safe_to_promote" | "needs_more_data";
  metadata?: Record<string, any>;
};

==================================================
TASK 8 — DO NOT ALLOW DIRECT SELF-MODIFICATION
==================================================

This is critical.

The live agent must NOT:
- rewrite its own prompt automatically
- change safety rules automatically
- absorb user advice as truth automatically
- modify validators automatically
- change programming doctrine directly from production interactions

The Global Learning Layer should only:
- collect
- aggregate
- suggest

Promotion into the core system should be:
- explicitly reviewed
- or tightly limited to low-risk categories with clear thresholds
