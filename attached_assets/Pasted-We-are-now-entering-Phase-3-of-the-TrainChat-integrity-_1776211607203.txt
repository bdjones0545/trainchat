We are now entering Phase 3 of the TrainChat integrity rebuild.

OBJECTIVE:
Unify the guest and paid user product pipeline so that every new user experiences the real TrainChat vibe-coding engine before the paywall, instead of a weaker, disconnected, or fake-light version.

Right now, the guest experience appears to be a separate or degraded path. That is a major conversion leak.

TrainChat’s growth depends on this truth:
The guest must experience the actual magic before being asked to pay.

From this point forward, the guest pipeline and paid pipeline should share the same core infrastructure wherever possible:
- same agent behavior
- same mutation engine
- same verification layer
- same truth layer
- same program-building experience
- same UI language
- same system-editing feel

The paywall should gate continuation, persistence depth, and premium access — not the core magic.

==================================================
PHASE 3 GOAL
==================================================

Refactor TrainChat so that an unauthenticated or non-paying user enters the same real vibe-coding workflow as a paid user, up until the exact moment the conversion wall is intentionally triggered.

The guest should not get:
- a fake preview engine
- weaker edit logic
- disconnected program generation
- generic chatbot behavior
- separate mutation logic
- reduced honesty or verification
- different UI states that make the product feel less real

The guest should get:
- the real agent
- the real program-building loop
- the real mutation logic
- the real verification system
- meaningful visible progress
- enough experience to become invested before gating

==================================================
CORE PRODUCT STANDARD
==================================================

Desired flow:

1. User enters TrainChat
2. User goes directly into the real agent experience
3. User starts describing goals and constraints
4. The actual program/system begins taking shape
5. The user sees visible, real edits and program structure
6. The user gets several meaningful interactions
7. The user becomes invested in the system being built
8. Only then does paywall/signup trigger
9. After signup/payment, the exact same system continues seamlessly

No product bait-and-switch.
No fake front half with a different backend.
No losing progress when the user converts.

==================================================
REQUIRED AUDIT + REFACTOR SCOPE
==================================================

Audit and refactor these areas:

A. ENTRY FLOW
Map exactly what happens when:
- first-time guest enters
- returning guest enters
- guest who already used some free interactions returns
- logged-out account user returns
- non-paying registered user enters
- paying user enters

Identify every route, gate, branch, fallback, and redirect.

I want to know:
- where guest and paid logic diverge
- whether separate agent paths exist
- whether different prompts/tools/stores are used
- whether different state models are used
- whether the guest path bypasses the real edit engine
- whether the guest path bypasses the real verification layer
- whether the guest path bypasses the real neural/personalization context

Then reduce unnecessary divergence.

B. AGENT PARITY
Ensure guest users use the same real agent orchestration stack as paid users wherever possible.

The guest must not receive:
- simplified “marketing AI”
- fake or canned edits
- weaker system modification behavior
- disconnected response logic
- different mutation capabilities before the paywall

Before gating, guest and paid should share:
- intent parsing
- edit orchestration
- clarification logic
- truth layer behavior
- mutation verification behavior
- coaching tone and system-editing logic

If limits are needed, those limits should be about session count or persistence depth, not intelligence quality.

C. PROGRAM STATE PARITY
Ensure guest users build against a real temporary/canonical program state, not a toy preview object.

Requirements:
- guest edits should affect a real structured program state
- guest changes should be visible and verifiable
- guest state should persist appropriately for the free experience window
- when the user signs up or pays, that state should be upgraded or attached, not discarded

I want a seamless handoff path from:
guest temporary state
→ authenticated user state
→ paid persistent state

No reset of the magic at conversion.

D. PAYWALL / GATING LOGIC
Refactor the paywall so it appears at the right moment:
after real value is demonstrated.

The gate should occur based on intentional product strategy such as:
- after X meaningful interactions
- after the first real program/system is generated
- after the user tries to continue beyond preview depth
- after save/export/continue requests
- after a defined conversion threshold event

Do not trigger the gate:
- before the user feels the product working
- on first input unless intentionally tested
- in a way that breaks momentum
- in a way that discards progress

Audit all gating triggers and remove premature interruption.

E. DEVICE ID / SESSION / AUTH TRANSITION
Audit device ID and guest identity handling deeply.

I want to know:
- how guest sessions are tracked
- how free interactions are counted
- how guest program state is stored
- how guest state is restored on return
- how logout affects device-based state
- whether device ID conflicts with account state
- whether account login causes lost guest progress
- whether auth state changes cause loops, reroutes, or agent interruption

Refactor so that:
- guest identity is stable enough for free-session continuity
- auth transitions are smooth
- device ID does not create routing bugs or overwritten state
- program state is merged or migrated safely when user signs up/logs in

F. CONVERSION CONTINUITY
When a guest signs up or pays:
- preserve the current conversation context where appropriate
- preserve the current program/system state
- preserve visible progress
- preserve the feeling that the build is continuing, not restarting

After conversion, the user should feel:
“I unlocked more of what I was already building”
not
“I got kicked into a different app”

Move toward a single shared product engine with layered entitlements:

SHARED CORE:
- agent orchestration

ENTITLEMENT LAYER:
