BUG 01 — Ambiguous “make it better” triggers REBUILD_PROGRAM

Problem:
Typing vague phrases like:
- “make it better”
- “improve it”
- “optimize it”
- “make this better”
- “make the program better”

currently triggers REBUILD_PROGRAM and saves a new program.

Expected:
These should ask a clarification question unless the user gives a specific target and direction.

Fix:
In execution-planner.ts and any intent/intent-family classifier used by buildExecutionPlan:

1. Add a vague improvement detector:
const VAGUE_IMPROVEMENT_PATTERNS = [
  /\bmake (it|this|the program|my program) better\b/i,
  /\bimprove (it|this|the program|my program)\b/i,
  /\boptimi[sz]e (it|this|the program|my program)\b/i,
  /\bmake (it|this) more effective\b/i,
  /\bupgrade (it|this|the program|my program)\b/i
]

2. If a vague improvement phrase is detected:
- If active program exists:
  action = ASK_CLARIFICATION
  clarificationReason = "vague_program_improvement"
  suggestedQuestion = "What would you like to improve — strength, endurance, explosiveness, recovery, session length, or a specific day?"
- If no active program exists:
  action = ASK_CLARIFICATION
  suggestedQuestion = "What would you like me to build or improve? For example: strength, speed, mobility, endurance, or recovery."

3. Do NOT allow the no-program branch to default vague improvement phrases to REBUILD_PROGRAM.

4. Do NOT generate structuredData for this clarification response.

Acceptance:
Typing “make it better”:
- with active program → asks clarification, no rebuild
- with no active program → asks clarification, no rebuild
- no new trainingSystemId created
- no “Rebuilding your training system” stage


BUG 02 — Mobility “Make Day 3 shorter and easier” fails with NO RECEIPT

Problem:
For a valid Mobility program, typing:
“Make Day 3 shorter and easier.”
returns failure even though target and intent are clear.

Expected:
Only Day 3 updates. Scope = session. systemEdit.applied = true.

Fix:
Trace and fix the mobility edit path:

1. Ensure resolveTargetFromRequest can resolve Day 3 for mobility systems.
Check:
- mobility sessions have valid dayIndex / sessionIndex / display order
- Day 3 maps to the third session in the active week
- focusMode mobility does not skip session lookup

2. Ensure “shorter and easier” maps to a valid edit intent/transformation:
- reduce_time
- lower_impact
- fatigue_management
- easier_session
- mobility_reduce_difficulty

3. If focusMode === "mobility" and message includes:
- shorter
- easier
- gentler
- less intense
- lower impact
- desk friendly
Then route to a session-scope modification when a day is specified.

4. Add a deterministic fallback before LLM edit failure:
If target day is resolved and intent is reduce_time/easier/lower_impact:
- apply session-level refinement directly
- reduce total volume/time
- preserve mobility exercise identity
- shorten holds/reps/sets where appropriate
- avoid converting mobility to strength prescriptions

5. Change failure handling:
Do not render “Agent Turn Report NO RECEIPT” to the user.
If receipt is missing, hide the debug card or show only the coach-safe failure message.

Acceptance:
Mobility program + “Make Day 3 shorter and easier.”
- Day 3 changes only
- no full rebuild
- systemEdit.applied = true
- Live Program Panel updates
- change log scope = session
- no visible “NO RECEIPT” debug label
