We have successfully completed anonymous deviceId authentication and real conversation creation for anonymous users.

Now implement the next phase:

AUTO-CREATE / AUTO-BOOTSTRAP THE CANONICAL ACTIVE TRAINING SYSTEM

OBJECTIVE:
Ensure that once an anonymous or authenticated user makes their first meaningful training-build request, TrainChat creates a real canonical training system record and sets it as the active editable system.

Right now:
- anonymous users are real users
- conversations are real
- session history is real
- auth works correctly across HTTP and HTTPS environments
- but /api/training-system/active returns 404 until a training system exists

That 404 is truthful, but now we need to eliminate this gap in the product experience.

==================================================
GOAL
==================================================

When a user sends their first real training-building request, the system should:

1. detect that no active training system exists
2. determine whether the message is a real build/program request
3. create a canonical training system automatically
4. create all required child records as needed
5. mark it as the active training system
6. return it consistently through the existing active-system flow
7. bind future edits to that same system

From that point on, the user should feel like they are editing a real live program.

==================================================
REQUIRED BEHAVIOR
==================================================

If /api/training-system/active is requested before any program exists:
- returning 404 is acceptable internally
- but once the user makes a meaningful build request, TrainChat should create the active system automatically

A “meaningful build request” includes messages like:
- Build me a 3-day strength program
- I want speed work for football
- Create a beginner fat loss plan
- Make me a 4-day hypertrophy split
- Build around dumbbells and bands

A generic greeting like:
- hi
- hello
- what can you do
should not create a system yet
