We are now running a focused TrainChat product polish sprint.

OBJECTIVE:
Close the final gap between “technically real” and “authentically feels like vibe coding for training.”

This is NOT an infrastructure rebuild.
Do NOT re-architect the mutation engine, SSE pipeline, verification system, undo system, or canonical program model unless absolutely necessary for one of the polish tasks below.

The infrastructure is already strong.

This sprint is about:
- trust
- ownership
- visibility of change
- first impression
- conversion continuity
- reducing “metered demo” signals
- making the realness impossible to miss

==================================================
PRIMARY GOAL
==================================================

Make TrainChat feel less like:
- a metered AI trial
- a guest mode
- a smart chatbot with visuals

And more like:
- a real training workspace
- a live editable system
- a collaborative coach-builder
- Cursor/Figma/Replit for training

SPRINT PRIORITIES
==================================================

Implement these in exact priority order unless you discover a dependency that forces a different sequence.

1. Fix free registered users so signup is always an upgrade

==================================================
TASK 1 — SIGNUP MUST NEVER DEGRADE THE EXPERIENCE
==================================================

Current known issue:
Registered free users get fewer messages than anonymous users.
This is a trust-destruction event and must be fixed first.

Audit and patch:
- planGating.ts
- any free/anonymous entitlement config
- any merge behavior carrying over or penalizing message counts
- any paywall messaging tied to these limits

Requirements:
- FREE_MESSAGE_LIMIT must be >= ANON_MESSAGE_LIMIT
- ideally free registered is a clear upgrade over anonymous
- signup must preserve momentum and feel beneficial
- if anonymous usage carries over, ensure it does not create a “you signed up and now have less” feeling
- if needed, reset or reframe counters on signup so it clearly feels like continuation or upgrade

Recommended outcome:
- anonymous users get enough to feel the product (e.g. 8)
- free registered users get at least 10 or 12, or another clearly better entitlement
- paid remains meaningfully better than both

Output expectation:
- explicitly state old limits and new limits
- explain how signup now behaves
- confirm there is no longer a downgrade path

==================================================
TASK 2 — REMOVE THE VISIBLE COUNTDOWN FROM FIRST IMPRESSION
==================================================

Current known issue:
The “8 free left” badge is visible immediately and makes the app feel metered from second 1.

Patch the UI so:
- no visible free-counter appears at first load
- user can enter the product without feeling they are racing a meter
- the limit is still enforced in the backend
- limit awareness is introduced only near the gate

Preferred UX:
- hide the top-right free message countdown entirely
- after the user is close to the limit (for example after message 6 of 8), show a subtle in-chat nudge or lightweight system message
- phrase it as preserving progress / saving their system, not consuming remaining credits

Examples of better language:
- “You’re close to the point where we’ll ask you to save your system.”
- “A couple more edits and we’ll help you save your build.”
- “You’ve built something real — next we’ll prompt you to save it.”

Do NOT use:
- “2 free left”
- “credits remaining”
- language that feels transactional or trial-like

Audit and patch:
- chat.tsx
- any badge components
- any top-nav free limit indicators
- anonymous / free user status indicators

==================================================
TASK 3 — ADD A PERSISTENT IN-CHAT CHANGE CONFIRMATION ARTIFACT
==================================================

Current known issue:
Real mutations happen, but the main proof lives in the right panel highlight and can be missed.

Implement a first-class in-chat mutation artifact that appears after every verified successful edit.

This should be a real product component, not just plain assistant text.

Requirements:
After a successful verified mutation, render a persistent SystemChangeCard (or equivalent) in the chat that includes:

- short title, e.g. “Program Updated”
- exact thing that changed
- where it changed (day/session/block if known)
- what it replaced or what was added/removed
- concise impact summary if available
- a CTA like “View in Program” or “Jump to Change”

Examples:
- “Day 2 updated: Bench Press → Incline Dumbbell Press”
- “Lower-body block updated: Added 1 set to squat variations”
- “Program adjusted: Added more speed work to Day 2”

Behavior:
- persists in the chat history
- survives scrolling
- is visually distinct from regular assistant messages
- clicking it opens the relevant panel/tab and scrolls to the changed item if possible
- should work on desktop and mobile
- should not disappear after a few seconds

It must be generated from the VERIFIED mutation result, not from raw intent.

Audit and patch:
- message rendering components
- message bubble / system update card logic
- mutation result → UI artifact mapping
- chat panel / live panel linking behavior

==================================================

Current known issue:

Remove “Anonymous” from all user-facing identity surfaces.

- identity framing should increase ownership

Audit and patch:
- left sidebar labels
- header identity labels
- session/history labels if applicable
- any paywall or empty-state copy that references anonymous/guest mode

TASK 5 — EXTEND AND REPLAY CHANGE VISIBILITY
==================================================

Current known issue:
The highlight animation is good but too easy to miss.

Improve mutation visibility in the Live Program panel.

Requirements:

Preferred UX:

- clicking it re-opens the relevant day/block if needed, scrolls to the changed area, and replays the highlight animation
- the panel header can optionally show a compact “Last updated: Bench Press → Incline DB Press” chip that persists for a while

Audit and patch:
- highlight timing constants
- changeTargets handling
- undo toast actions
- panel header/status chip logic
- scroll + replay behavior
