UX FIX — DO NOT HIDE THE ACTIVE PROGRAM WHEN THE AGENT IS WAITING FOR A REPLY DURING A NEW BUILD

CONTEXT:
A Speed program already exists in the database and is active.

However, when the user starts a new Speed build conversation, the chat enters a “new build” flow and the right panel hides the existing active program while the agent waits for more input.

This makes it seem like:
- no program exists
- the build failed
- the user lost their program
- the system is broken

The user did not realize they were expected to reply to continue the build.

==================================================
PRIMARY GOAL
==================================================

Make the UI clearly distinguish between:
1. the existing active program
2. a new build draft that is waiting for user input

Do not hide the active program in a confusing way.

==================================================
TASK 1 — DEFINE TWO DISTINCT STATES
==================================================

Add explicit UI state separation between:

A. ACTIVE PROGRAM
- saved in DB
- currently usable
- should remain visible

B. IN-PROGRESS BUILD DRAFT
- currently being created in chat
- incomplete
- waiting for user input or completion

These must not be visually conflated.

==================================================
TASK 2 — KEEP THE ACTIVE PROGRAM VISIBLE
==================================================

If a focus already has an active program in DB and the user starts a new build conversation:
- keep the active program visible in the right panel
OR
- show a clear toggle between:
  - Current Program
  - New Draft

Do NOT replace the active program view with an empty/hidden panel while waiting for user input.

==================================================
TASK 3 — ADD A “WAITING FOR YOUR REPLY” STATUS
==================================================

When the agent asks a follow-up question before it can complete the build, show a clear status message in the UI:

Examples:
- Waiting for your reply to finish this build
- Your current Speed program is still active
- Reply to continue building the new draft

This must be obvious on mobile.

==================================================
TASK 4 — ADD A NEW BUILD DRAFT CARD / BANNER
==================================================

If a new build is in progress, show a compact draft status card in the right panel or near the top of chat:

Example:
NEW SPEED DRAFT IN PROGRESS
Waiting for: profile details / equipment / training days
[Continue Build]

This tells the user the system is paused, not broken.

==================================================
TASK 5 — DO NOT ENTER “EMPTY STATE” IF AN ACTIVE PROGRAM EXISTS
==================================================

If an active program exists for the current focus, never show an empty or hidden right panel just because a draft build is unfinished.

The active program should remain the fallback visible asset until the new draft is completed and explicitly promoted.

==================================================
TASK 6 — PROMOTION RULES
==================================================

Once the new build is completed successfully:
- it can replace/promote over the old active program if that is the intended behavior
- until then, the old active program stays visible and active

Do not prematurely swap active context to an incomplete draft.

==================================================
TASK 7 — CHAT COPY IMPROVEMENT
==================================================

When the agent asks a clarifying question during a build, make the language more explicit:

Bad:
“I’ll map the full exercise selection once you complete your profile.”

Better:
“I’m waiting for one more reply from you before I can finish this new Speed program. Your current Speed program is still available on the right.”

==================================================
TASK 8 — AUDIT DRAFT VS ACTIVE STATE LOGIC
==================================================

Trace:
- when a chat enters new build mode
- when the right panel switches views
- when active DB program is hidden
- when draft becomes active

Return exact logic and fix any premature handoff.

==================================================
TASK 9 — VALIDATION
==================================================

Test:
1. User already has active Speed program
2. User starts a new Speed build
3. Agent asks a follow-up question
Expected:
- active program remains visible
- draft status is visible
- user clearly sees that a reply is needed
- no confusion that the system is broken

==================================================
TASK 10 — OUTPUT REQUIRED
==================================================

Return:
1. how active vs draft state is now separated
2. how the right panel behaves during incomplete builds
3. what waiting-for-reply UI was added
4. what chat copy was improved
5. confirmation that existing active programs no longer disappear confusingly

==================================================
FINAL STANDARD
==================================================

If the agent needs another reply to complete a new build, the user must clearly understand that:
- the build is waiting on them
- their current program still exists
- nothing is broken
- replying will continue the draft
