Add a SPORT LANGUAGE NORMALIZATION SYSTEM to TrainChat so the app can understand slang, shorthand, nicknames, role words, and informal sport references such as “boxer,” “pickle,” “flag,” “lax,” “pitcher,” etc., and safely map them to canonical sport profiles.

GOAL
Users should not need to type exact formal sport names.
The system should understand natural language sport identity and convert it into canonical sport/profile IDs that the ranking and programming engine can use.

CORE PRINCIPLE
Separate:
1. what the user said
2. what the system thinks it means
3. the canonical sport/profile ID used by the app

Do NOT rely on exact string matching only.

==================================================
PHASE 1 — CREATE SPORT LANGUAGE NORMALIZER
==================================================

Create a new file:
- sport-language-normalizer.ts

Define:

type SportResolution = {
  rawInput: string
  normalizedText: string
  matchedAlias?: string
  canonicalSportId?: string
  canonicalRoleId?: string
  confidence: number
  ambiguity?: string[]
  resolutionType: "exact" | "alias" | "contextual" | "ambiguous" | "none"
}

Create:

function resolveSportLanguage(input: {
  text: string
  currentProfile?: {
    primarySport?: string
    secondarySports?: string[]
    positionOrRole?: string
  }
  availableSports: string[]
}): SportResolution

Requirements:
- normalize casing
- strip punctuation safely
- support singular/plural variants
- support slang and shorthand
- support role-based words
- support common misspellings
- return confidence score
- do not crash on unknown phrases

==================================================
PHASE 3 — ADD CONTEXTUAL DISAMBIGUATION
==================================================

Create helper logic:

function disambiguateSportAlias(input: {
  normalizedText: string
  candidateMatches: SportResolutionCandidate[]
  currentProfile?: {
    primarySport?: string
    secondarySports?: string[]
    positionOrRole?: string
  }
}): SportResolution

Rules:
- If current profile already contains cricket, “bowler” should prefer cricket_bowler
- If current profile already contains bowling, “bowler” should prefer bowling
- If text includes “cricket bowler”, resolve directly to cricket_bowler
- If text includes “wicketkeeper” or “keeper” with cricket nearby, resolve to cricket_wicketkeeper
- If text includes “goalkeeper” or soccer nearby, do not route to cricket
- If ambiguity remains, return resolutionType = "ambiguous" and list candidate sports

Do not silently force low-confidence ambiguous matches.

==================================================
PHASE 4 — SUPPORT PHRASE EXTRACTION FROM CHAT
==================================================

The system should detect sport identity from natural sentences, not just isolated words.

Examples it should parse:
- “I’m a boxer.”
- “I play pickle 3 times a week.”
- “I do flag and run track.”
- “Former wrestler, now getting into MMA.”
- “I’m a cricket bowler.”
- “Mostly a bowler, but I also bat.”
- “I’m a pitcher.”
- “I coach lax.”

Create helpers that can scan full free-text input and extract likely sport mentions.

==================================================
PHASE 5 — STORE RAW + CANONICAL
==================================================

When a sport is detected or selected, store both:
- rawSportInput
- canonicalSportId
- canonicalRoleId if applicable
- confidence

This is important for:
- future explainability
- debugging
- improving the alias system later

==================================================
PHASE 6 — SAFE FALLBACK BEHAVIOR
==================================================

If no confident match is found:
- do not break flow
- leave sport unresolved
- fall back to current behavior
- log the unknown phrase for future alias expansion

If ambiguous:
- mark it as ambiguous
- do not overwrite an existing strong sport identity unless confidence is high

==================================================
PHASE 7 — EXPLAINABILITY
==================================================

Return useful debug/explain fields such as:
- rawInput
- matchedAlias
- canonicalSportId
- confidence
- ambiguity candidates

This should let the agent later say things like:
- “I interpreted ‘pickle’ as pickleball.”
- “I interpreted ‘boxer’ as boxing.”
- “I found ‘bowler’ ambiguous between bowling and cricket bowler, so I preserved your current cricket profile.”
