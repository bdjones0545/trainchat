TrainChat — Settings Intelligence Control Center Sprint

Objective:
Transform Settings from a generic account/preferences page into the control center for the user's relationship with Atlas.

Core Principle:
Settings should feel like:
- where the athlete shapes how Atlas coaches,
- what Atlas remembers,
- how aggressively Atlas adapts,
- and how proactive/intelligent the system becomes.

The page should reinforce:
persistent athlete intelligence.

==================================================
P0 — INFORMATION ARCHITECTURE REFRAME
==================================================

Rename route:
- /billing → /settings

Reorganize sections in this order:

1. Athlete Identity
2. Atlas Intelligence
3. Coaching Behavior
4. Adaptation + Recovery
5. Memory + Privacy
6. Notifications + Coaching Presence
7. Subscription + Usage
8. Safety + Constraints
9. Danger Zone

Goal:
The page should feel identity-first, not billing-first.

==================================================
P0 — COACH MEMORY VISIBILITY
==================================================

Integrate CoachMemoryPanel directly into Settings.

Requirements:
- Add dedicated "What Atlas Knows About You" section
- Reuse existing memory categories and confidence badges
- Add lastUpdated timestamp to every memory
- Add source label prominence
- Add empty states for categories with no memories

New actions:
- Edit memory
- Delete memory
- Mark as resolved
- Downgrade confidence

Behavior:
- “Resolved” should preserve history but stop active prompt injection
- Memories marked resolved remain visible in a subdued style
- Resolved injury/pain memories should move into a “Resolved Constraints” subsection

==================================================
P0 — INJURY + CONSTRAINT LIFECYCLE
==================================================

Create explicit safety state system.

Add states:
- Active
- Monitor
- Resolved

Apply to:
- injuries
- pain_pattern memories
- movement restrictions

UI:
- Dedicated “Safety + Constraints” section
- Separate from normal preferences
- High visual prominence
- Timeline-style cards preferred

New actions:
- Mark healed
- Reduce restriction level
- Remove permanently

Resolver behavior:
- Active → full prompt enforcement
- Monitor → soft caution injection
- Resolved → excluded from adaptation logic but retained historically
