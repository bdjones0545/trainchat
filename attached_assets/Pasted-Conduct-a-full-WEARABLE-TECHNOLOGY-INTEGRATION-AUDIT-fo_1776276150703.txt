Examples:
- Apple Health HRV vs WHOOP recovery vs Oura readiness

- step counts across providers
- workouts with different schemas

- date boundaries
- metric units

I want to know:
- is there a normalized internal model?
- how are incompatible provider schemas handled?
- are there misleading equivalencies being made?
- are provider-specific metrics preserved or oversimplified?

G. STALENESS / MISSING DATA HANDLING
This is critical.

Audit:
- how the system detects stale wearable data
- what happens when data is missing

- whether the UI or agent still speaks as if the data is current
- whether adaptation logic incorrectly assumes data exists

- what happens if today’s recovery data is missing?
- what happens if the provider connection broke yesterday?
- does the app fall back gracefully or hallucinate certainty?

Audit whether wearable data is actually used by:
- adaptation engine
- readiness logic

- fatigue management

I. CONFLICT / OVERRIDE LOGIC

- user-reported readiness

I want to know:

- does wearable data overrule everything?

- connected vs disconnected state
- last sync timestamp
- sync error visibility
- stale data warnings

K. WEBHOOKS / PROVIDER PUSH EVENTS (IF USED)
If any providers use webhooks or push sync:

- daily summaries
- readiness/recovery records

I want to know:

==================================================
MANDATORY TEST SCENARIOS
==================================================

Audit and trace these exact scenarios:

1. User connects a wearable for the first time
- auth succeeds
- provider account links to correct user
- token stored correctly
- initial sync runs correctly

2. Daily sync succeeds
- new metrics arrive
- normalization works
- last sync updates
- app reflects fresh data

3. Daily sync fails
- error is recorded
- stale data is detectable
- app does not pretend data is current

4. Access token expires
- refresh flow works
OR
- user is prompted correctly to reconnect
- no silent broken integration

5. Provider data is missing for a day
- adaptation logic behaves safely
- no false readiness certainty

6. User disconnects provider
- tokens revoked/removed appropriately
- UI updates correctly
- system stops using stale wearable data as if live

7. Wearable data conflicts with user performance trend
- system handles conflict intelligently

8. Wearable-connected user gets an adaptation recommendation
- verify whether wearable data actually changed the recommendation

9. Duplicate sync / repeated provider events
- no duplicate daily records
- no corrupted summaries

10. Anonymous / free / paid transitions (if relevant)
- wearable connection remains correctly linked to the user identity
- no broken ownership or lost integration
