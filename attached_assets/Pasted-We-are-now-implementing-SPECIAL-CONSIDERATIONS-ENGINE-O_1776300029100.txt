We are now implementing:

SPECIAL CONSIDERATIONS ENGINE

OBJECTIVE:
Allow TrainChat to intelligently handle users with special considerations — including older adults, Parkinson’s, neurological limitations, balance issues, frailty, major mobility limitations, and similar cases — by switching from normal library-first programming to a safer OpenAI-guided construction mode with hard guardrails.

This should expand TrainChat beyond standard athletic/general fitness programming without forcing these users into a rigid exercise database that may not contain the right supported, simplified, or symptom-aware options.

IMPORTANT:
This is NOT a medical diagnosis engine.
This is NOT rehab or treatment.
This is a safer programming mode for users with meaningful limitations or special considerations.

==================================================
CORE PRINCIPLE
==================================================

Normal users:
- library-first
- deterministic exercise selection
- standard programming engines

Special-consideration users:
- OpenAI-guided construction allowed beyond the normal exercise DB
- stricter safety constraints
- more conservative progression
- validator-enforced output quality
- lower fall risk / lower coordination risk / lower complexity by default

The system should not become reckless or hallucination-prone.
It should become more flexible while remaining more conservative.

==================================================
PRIMARY GOAL
==================================================

When TrainChat detects a special-consideration context, it should:
1. switch into Special Considerations Mode
2. let OpenAI construct safer, more context-specific programming beyond the normal exercise library if needed
3. still apply strong safety and structure validation before output is accepted

==================================================
TASK 3 — DEFINE SAFETY-FIRST PROGRAMMING RULES
==================================================

In special_considerations mode, enforce rules like:

- prioritize stability and support
- prefer lower complexity
- reduce coordination demands unless clearly appropriate
- minimize fall risk
- bias toward supported or constrained environments
- use conservative volume and intensity
- slower progression
- fewer exercises per session
- simple transitions between movements
- reduce unnecessary novelty
- avoid explosive/reactive work by default unless clearly appropriate
- avoid max-effort loading
- avoid dense fatigue-driven circuits by default
- emphasize movement quality, tolerance, confidence, and repeatability

For Parkinson’s / neurological / balance-sensitive users:
- bias toward gait, balance, postural control, sit-to-stand patterns, supported lower-body work, manageable trunk work, and simple strength patterns

For older adults:
- bias toward strength for function, chair-supported work, balance confidence, controlled mobility, and manageable conditioning

- seated or supported trunk rotation

- low-complexity mobility/resilience circuits

IMPORTANT:

TASK 7 — ADD SPECIAL-CONSIDERATIONS VALIDATOR
==================================================

Create a validation layer specifically for special_considerations mode.

Validator should check for issues such as:
- too many exercises in one session
- too much total volume
- excessive coordination demands
- unsupported balance challenges
- high fall-risk patterns without support
- unnecessary explosive work
- high-intensity conditioning too early
- unrealistic transitions for limited users

Example checks:

- if older adult / frailty and session density is too high → fail

- if special_considerations mode and language implies max effort → fail

Output should be retried or corrected if validator fails.

==================================================
TASK 8 — ADD CLARIFICATION RULES FOR HIGH-RISK CONTEXT
==================================================

Unlike normal vibe-coding mode, in special_considerations mode the system should ask clarification more readily when safety-relevant details are missing.

Examples where clarification is appropriate:
- severe balance limitation but no support environment mentioned
- Parkinson’s mentioned with unclear current function
- stroke history with unclear mobility level
- joint replacement with unclear movement tolerance
- significant pain / instability language

But do NOT over-ask for trivial details.
Still be useful.

Examples:
- “Do you have a stable chair or counter available for support?”
- “Are you able to get up and down from a chair comfortably?”
- “Should I assume we need low-balance-risk options only?”
