We need to add a complete FORGOT PASSWORD / RESET PASSWORD flow to TrainChat so users can securely reset their password from the sign-in screen.

OBJECTIVE:
Users need a clear, secure, production-ready password reset experience.

The system should allow a user to:
1. click “Forgot password?”
2. enter their email
3. receive a branded reset email
4. click a secure reset link
5. set a new password
6. sign in successfully

This must work cleanly with the new email infrastructure.

==================================================
PRIMARY GOAL
==================================================

Implement a full password reset system that is:
- secure
- user friendly
- mobile friendly
- branded
- production ready

==================================================
TASK 1 — ADD “FORGOT PASSWORD?” TO AUTH UI
==================================================

On the sign-in screen:
- add a clear “Forgot password?” action
- place it in an intuitive spot near the password field / sign-in action
- make sure it is easy to tap on mobile

When clicked:
- open a dedicated forgot-password form or route
- ask for the user’s email only

==================================================
TASK 2 — CREATE FORGOT PASSWORD REQUEST FLOW
==================================================

Create an endpoint / backend flow that accepts:
- email address

Behavior:
- always return a safe generic success response
- do NOT reveal whether the email exists
- if the email exists, create a secure reset token and send the email
- if the email does not exist, still show the same success response

Example user-facing response:
“If an account exists for that email, we’ve sent password reset instructions.”

==================================================
TASK 3 — SECURE TOKEN GENERATION
==================================================

Generate a secure password reset token that is:
- cryptographically random
- time-limited (recommended: 30–60 minutes)
- single-use
- tied to the user account

Store only what is needed securely.
Prefer hashed token storage if cleanly possible.

Suggested fields if using a DB table:
- id
- userId
- tokenHash
- expiresAt
- usedAt
- createdAt

==================================================
TASK 4 — SEND RESET EMAIL USING THE NEW EMAIL SYSTEM
==================================================

Use the new branded email layer (SendGrid implementation) to send a password reset email.

Create a new email template:
- Password reset request

Email should include:
- clear explanation
- secure reset button/link
- expiration notice
- support guidance if the user didn’t request it

Example subject lines:
- Reset your TrainChat password
- TrainChat password reset request

The email should match TrainChat branding and be mobile friendly.

==================================================
TASK 5 — CREATE RESET PASSWORD PAGE / ROUTE
==================================================

Create a route like:
- /reset-password?token=...

When user opens the link:
- validate token
- if valid, show reset password form
- if invalid or expired, show clean error state with option to request another reset

Reset form fields:
- new password
- confirm new password

==================================================
TASK 6 — PASSWORD VALIDATION
==================================================

Add clear password rules, for example:
- minimum 8 characters
- recommended strength checks if easy
- confirm password must match

Validation should be friendly and inline.

==================================================
TASK 7 — COMPLETE RESET FLOW
==================================================

When user submits a new password:
- verify token
- verify not expired
- verify not already used
- update password securely
- mark token as used
- invalidate any reused token path
- optionally revoke existing sessions if appropriate
- redirect user to sign in with success message

Success message example:
“Your password has been reset. You can now sign in.”

==================================================
TASK 8 — SECURITY REQUIREMENTS
==================================================

Must include:
- generic response on forgot-password request
- no user enumeration
- token expiration
- single-use tokens
- secure password hashing
- rate limiting on forgot-password endpoint if possible
- clear audit logging in development

Optional but recommended:
- invalidate old reset tokens when a new one is issued
- revoke active sessions on password reset if architecture supports it

==================================================
TASK 9 — MOBILE UX
==================================================

Ensure:
- forgot-password page looks good on iPhone/mobile Safari
- reset-password page is clean and easy to use
- keyboard does not hide inputs/buttons
- success/error states are polished

==================================================
TASK 10 — DEV / QA LOGGING
==================================================

Add development logs like:

[PasswordResetRequest]
[PasswordResetEmailSent]
[PasswordResetTokenValidated]
[PasswordResetCompleted]

Do not log raw tokens in plaintext in production.

==================================================
TASK 11 — TEST REQUIRED
==================================================

Verify all of these:

1. Existing user enters email
Expected:
- generic success message
- reset email sent

2. Nonexistent email enters email
Expected:
- same generic success message
- no enumeration leak

3. Valid reset link opens reset form
Expected:
- can set new password

4. Expired token
Expected:
- clean expired message
- prompt to request another reset

5. Used token
Expected:
- rejected cleanly

6. After reset, user can sign in with new password
Expected:
- success

7. Old password no longer works
Expected:
- rejected

==================================================
TASK 12 — OUTPUT REQUIRED
==================================================

Return:
1. UI changes added
2. backend endpoints/routes added
3. DB table / token storage used
4. email template added
5. security protections included
6. test results

==================================================
FINAL STANDARD
==================================================

TrainChat should have a complete forgot-password flow that feels premium and trustworthy.

Users should be able to reset their password securely through branded email, without leaking account existence and without introducing insecure token behavior.
