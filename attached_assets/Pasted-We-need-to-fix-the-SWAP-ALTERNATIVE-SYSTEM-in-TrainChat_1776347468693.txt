We need to fix the SWAP ALTERNATIVE SYSTEM in TrainChat.

OBJECTIVE:
Restore usable swap alternatives for exercises when the current cluster lookup returns empty, especially for cases like Box Jump where the exercise has a clusterId but every member of that cluster is inactive.

Right now the system often shows:
“No cluster alternatives found. Use Full Edit to swap manually.”

The audit found that this is happening for most of the library because:
- many exercises have cluster IDs
- but their clusters have zero active members
- the code only falls back to movement-pattern search when clusterId is null
- if clusterId exists but clusterMembers returns empty, the fallback is never reached
- frontend also hides real API errors behind the same empty-state message

We need to fix this now.

==================================================
PRIMARY GOALS
==================================================

1. If a cluster lookup returns zero active members, automatically fall back to movement-pattern search
2. Distinguish “no alternatives found” from “couldn’t load alternatives”
3. Restore high-value clusters like plyometric_jump
4. Prepare the swap path for an intelligent fallback when deterministic results are still empty

==================================================
TASK 1 — PATCH EMPTY-CLUSTER FALLBACK LOGIC
==================================================

Audit getSwapCandidates() in exercise-service.ts.

Current broken behavior:
- findExerciseByName() returns an exercise with clusterId
- getClusterMembers(clusterId, exerciseName) returns []
- because clusterId exists, code never reaches getByMovementPattern()
- returns empty array immediately

Fix this logic.

New desired behavior:
- if exercise is missing → fallback to movement pattern search if possible
- if clusterId is missing → fallback to movement pattern search
- if clusterId exists but clusterMembers.length === 0 → fallback to movement pattern search
- only return empty if BOTH cluster path and movement-pattern fallback fail

This is the highest-priority fix.

==================================================
TASK 2 — PRESERVE EXISTING FILTERING, BUT ONLY AFTER RESULTS EXIST
==================================================

Make sure equipment/injury/neural demand/time cost filters still apply,
but do not let them block fallback from even running.

Expected logic:
1. find exercise
2. try cluster alternatives
3. if cluster alternatives empty → try movement-pattern alternatives
4. then filter/rank
5. return candidates

==================================================
TASK 4 — PATCH FRONTEND EMPTY VS ERROR STATES
==================================================

In system.tsx / swap UI flow:

Current behavior:
- API error
- timeout
- empty results
all produce:
“No cluster alternatives found”

This is wrong.

Add distinct frontend states:

A. swapCandidates loaded, empty
→ “No alternatives found. Use Full Edit to swap manually.”

B. swap lookup failed / network error / backend error
→ “Couldn’t load alternatives. Try again.”

Do NOT collapse these into the same UI message.
