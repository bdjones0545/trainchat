Now perform a full post-migration validation sweep.

GOAL
We do not want more model changes right now.
We want to validate that TrainChat still behaves correctly after the migration and identify any regression risks, schema drift, tone drift, verbosity drift, or latency concerns.

TASK
Audit the full AI surface area and validate each migrated role against its expected behavior.

VALIDATION CATEGORIES

1. CORE CHAT VALIDATION
Target file:
- lib/ai.ts

Validate:
- the TrainChat agent still feels like TrainChat
- responses remain specific, useful, and high quality
- no over-explaining
- no weird generic AI tone

Check for:
- prompt compatibility issues
- higher verbosity than desired
- changes in decisiveness
- changes in formatting style
- unexpected refusal or hedging behavior
- token/latency concerns if visible in logs

2. ROUTING VALIDATION
Target file:
- lib/guestChat.ts

Validate:
- the lightweight guest routing step still classifies correctly
- structured routing output still parses cleanly
- no enum drift
- no JSON shape drift
- no branching regressions caused by different model output behavior

Check:
- markdown fences
- accidental prose around JSON
- missing keys
- malformed route decisions
- silent fallback behavior

3. GUEST RESPONSE VALIDATION
Target file:
- lib/guestChat.ts

Validate:
- guest-facing outputs still feel compelling

4. PROGRAM GENERATION VALIDATION

- fallback behavior

B. EDIT_ENGINE
