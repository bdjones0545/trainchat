We need to connect the new Block Variation Engine and the existing Exercise Variation / Slot Ranking system into one unified programming engine. Right now these systems risk living beside each other instead of driving each other. The result must be that the chosen block archetype materially changes which exercises win, how slots behave, how days feel, and how the final program differs from other outputs.

GOAL
Make block variation actively drive exercise variation.

CORE PROBLEM
Right now the system can still converge into “same program, different exercises” or worse, a mismatch where the chosen block identity says one thing but exercise selection behaves like another block. We need the block archetype, phase, split architecture, and day template to shape exercise ranking directly.

PRIMARY OBJECTIVES

1. Connect block variation to exercise ranking
2. Make slot behavior block-aware and phase-aware
3. Ensure exercise selection reflects block identity, not just generic slot fit
4. Add audit logging so we can prove why each exercise won
5. Prevent stale outputs by combining block-level novelty with exercise-level novelty

ARCHITECTURAL PRINCIPLE

Block engine sets the context.
Exercise engine obeys the context.

Do NOT let exercise ranking bypass block state.
Do NOT let slots use raw generic ranking without the derived block context.
Use single-source-of-truth principles.

SIMILARITY + ANTI-STALE INTEGRATION

Integrate exercise-level novelty with block-level novelty.

We already want block/split similarity checks, but now also add:
- same top lift repetition penalty
- same top 2 movement family pattern penalty by day
- same assistance cluster penalty
- same elastic/power cluster penalty
- same trunk/conditioning finisher repetition penalty

When generating a new program:
1. choose block archetype
2. choose split architecture
3. derive ProgramContextProfile
4. rank exercises per slot with block-aware scoring
5. compute program fingerprint
6. compare against recent program fingerprints
7. if too similar, try re-ranking with stronger novelty pressure or use next-best block/split candidate
8. log why fallback happened

Do NOT randomize blindly.
