PART 2 — CREATE A SPEED-SPECIFIC SYSTEM PROMPT
==================================================

Build a dedicated Speed system prompt, not a lightly modified Strength prompt.

It must encode:
- sprint / speed coaching language
- speed block archetypes
- speed session grammar
- movement families
- week progression logic
- full recovery rules
- quality-over-fatigue principles
- no strength bias

The prompt should clearly describe that this is a speed / footwork system, not a lifting program.

==================================================
PART 3 — BUILD A STRUCTURED SPEED ARCHITECTURE BRIEF
==================================================

Create a speed-specific architecture brief object that is injected into the OpenAI call before generation.

It must include:
- selected speed block archetype
- current week role
- session identities
- movement family requirements
- drill ordering rules
- volume / contact expectations
- conditioning rules
- progression intent

This should be as explicit as Strength’s architecture layer.

==================================================
PART 4 — DEFINE A SPEED RESPONSE CONTRACT
==================================================

The OpenAI output must conform to a clear structure.

Define a speed-native response contract that can be parsed deterministically into ProgramStructure.

It should include:
- program title
- program summary
- days
- sessions
- exercises/drills
- sets / reps / distance / rest as appropriate
- coaching notes
- week logic if required

Do NOT rely on loose prose output.
Speed generation needs a structured contract.

==================================================
PART 5 — ADD SPEED-SPECIFIC PARSING + VALIDATION
==================================================

Build or strengthen a speed-specific parser / validator.

Validation must ensure:
- focusMode = speed
- session identities are speed-native
- exercises are from speed families
- no strength-flavored session names like:
  - Lower Strength
  - Upper Push
  - Hypertrophy
- proper sprint / footwork / COD / reactive language appears
- structure is complete enough to persist

If OpenAI returns weak/invalid output:
- repair if possible
- otherwise reject cleanly

==================================================
PART 6 — ADD SPEED-SPECIFIC REPAIR LAYER
==================================================

If the OpenAI response is close but imperfect, add a repair layer before rejection.

Examples:
- missing session description → derive from architecture brief
- weak title → rewrite as speed-native title
- one bad strength-flavored session label → repair from selected speed session identity
- incomplete rest / volume field → fill from speed defaults

Important:
Repair should preserve the real speed engine and not become a hidden fallback builder.

==================================================
PART 7 — ENSURE NO STRENGTH BLEED IN OPENAI OUTPUT
==================================================

Add a final anti-bleed validation pass.

Reject or repair if the Speed output contains:
- strength titles
- hypertrophy language
- upper/lower lift splits

- it should operate inside the speed engine’s architecture, block structure, and family constraints
