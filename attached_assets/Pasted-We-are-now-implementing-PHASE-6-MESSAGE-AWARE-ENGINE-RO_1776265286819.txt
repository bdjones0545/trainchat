We are now implementing:

PHASE 6 — MESSAGE-AWARE ENGINE ROUTING LAYER

OBJECTIVE:
Fix the final integration gap in TrainChat by making all programming engines respond to the user’s LIVE message, not just stored onboarding/profile fields.

This is not a new intelligence phase.
This is the routing layer that finally unlocks the intelligence already built in Phases 1–5.

==================================================
CORE PROBLEM
==================================================

Right now, TrainChat has strong programming engines for:

- conditioning / energy systems
- power / speed
- sport-specific architecture
- advanced progression / block periodization
- re-entry / return-to-training

But these engines are still being triggered primarily from stored profile context, such as:
- profile.trainingGoal
- stored sport
- stored onboarding selections

This causes a major product failure:

A user can ask for something NEW in the live chat —
for example:
- “Add more speed work”
- “I’m in-season now”
- “I haven’t trained in 8 months”
- “Make this more soccer-specific”
- “Build my conditioning”

—and the system may fail to activate the correct engine, because the trigger logic is reading stale profile context instead of the current message.

This makes the product feel less adaptive than it actually is.

==================================================
CORE PRINCIPLE
==================================================

A true AI performance architect must adapt to what the user is asking for RIGHT NOW.

The user’s live message must be a first-class routing signal.

Stored profile context is useful, but it must be secondary to the current request.

==================================================
PRIMARY GOAL
==================================================

Refactor TrainChat so that every engine-routing decision is based on:

1. the user’s live message
2. the user’s stored profile
3. resolved priority rules between them

The system should use BOTH sources intelligently, but the live message must be able to override or reshape the profile-driven default.

==================================================
REQUIRED ARCHITECTURE
==================================================

Build a routing layer where all engine trigger functions receive both:

- userMessage
- profileContext

instead of only profile-derived fields

Examples of the kind of API shape I want:

needsConditioningContext(userMessage, profile)
needsPowerContext(userMessage, profile)
needsSpeedContext(userMessage, profile)
needsSportSpecificContext(userMessage, profile)
needsReEntryContext(userMessage, profile)
detectSeasonContext(userMessage, profile)

Use existing naming where appropriate, but the behavior must match this standard.

2. EXPLICIT LIVE MESSAGE REQUEST
3. SPORT-SPECIFIC CONTEXT
4. SEASON CONTEXT
5. PROFILE DEFAULT GOAL / PROFILE DEFAULT SPORT
6. GENERAL BASE PROGRAMMING RULES

Examples:

If profile says:
- strength

But live message says:
- “Add more speed work”

Then:
- speed engine must fire
- profile should inform background context, not override the live request

If profile says:
- off-season football

- “I’m in-season now”

- in-season modulation must override stored off-season assumptions

If user says:
- “I haven’t trained in 8 months”

- re-entry logic must override aggressive strength or performance defaults

Build this resolution cleanly and explicitly.
