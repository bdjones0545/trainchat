We already have a language system in TrainChat. This is NOT a net-new language feature build. This is a system upgrade.

GOAL
Upgrade the existing language system so it can handle much broader English coaching language, map user phrasing more reliably into programming intent, and respond appropriately across vague, direct, slangy, emotional, and conversational inputs.

CORE PROBLEM
The current language system likely works for many straightforward requests, but we need to harden and expand it so the agent can better interpret real-world English at scale.

The goal is NOT to literally “cover all English.”
The goal is to make the existing language system robust across broad English input patterns relevant to:
- training goals
- program creation
- program modification
- recovery feedback
- pain/constraint updates
- vibe-coding style edits
- preference language
- natural conversational follow-ups

PRIMARY OBJECTIVES

1. Audit the current language system
2. Identify current strengths, blind spots, and over-reliance on narrow patterns
3. Expand concept coverage for broad English coaching language
4. Improve mapping from user language → structured programming intent
5. Improve ambiguity detection, confidence scoring, and conflict detection
6. Preserve current working behavior while upgrading interpretation quality
7. Add DEV audit visibility so we can see what the system thinks the user meant

IMPORTANT
This is an UPDATE to the existing language system.
Do not replace working logic blindly.
Do not create a giant brittle phrase list.
Do not hardcode endless if/else branches.
Enhance the current architecture cleanly.

PHASE 1 — AUDIT THE CURRENT SYSTEM

First inspect the existing language understanding pipeline and document:

- where user text enters the system
- how it is normalized
- how intent is currently extracted
- how goals/constraints/preferences are currently mapped
- whether synonyms/slang are handled
- whether ambiguity/confidence are tracked
- whether structured intent objects already exist
- where the system passes interpreted meaning into the program engine
- where interpretation is too shallow or too brittle
- where surface-word matching is overused
- where existing logic can be extended instead of replaced

Print a concise audit summary before major code changes:

2. current structured outputs
3. current limitations
4. best integration points for the upgrade

REQUIRED UPGRADE APPROACH

Do not solve this with one giant keyword list.

Instead extend the current system into layered interpretation:

LAYER 1 — NORMALIZATION
Improve preprocessing and concept normalization for:
- typo tolerance
- plural/singular normalization
- phrase cleanup
- synonym grouping
- slang grouping
- fuzzy phrase mapping

Examples:
- explosive / pop / springy / bounce / game-speed
- smoked / cooked / beat up / fried
- grindy / heavy / sluggish
- simpler / easier / less complicated

LAYER 2 — STRUCTURED EXTRACTION
Upgrade the system so a single message can extract multiple categories at once:
- requestType
- primaryGoal
- secondaryGoals
- requestedChanges
- preserveInstructions
- constraints
- stylePreferences
- recoveryState
- equipmentConstraints
- scheduleConstraints
- ambiguityFlags
- confidenceScore

If a structured intent object already exists, extend it instead of replacing it.

Ideal shape should be something like:
AgentIntentProfile = {
  requestType,
  primaryGoal,
  secondaryGoals,
  requestedChanges,
  preserveInstructions,
  constraints,
  stylePreferences,
  recoveryState,
  equipmentConstraints,
  scheduleConstraints,
  ambiguityFlags,
  contradictions,
  confidenceScore,
  sourceUtterance,
}

LAYER 3 — INTERPRETATION / PROGRAM TRANSLATION
Upgrade the mapping from language meaning into programming meaning.

Examples:
“more athletic”
→ increase power / movement quality / performance bias
→ reduce bodybuilding isolation bias
→ maybe increase elastic / rotational / speed-support exposure depending on context

“less grindy”
→ reduce high-fatigue bilateral grinding
→ reduce excessive density
→ increase velocity intent or cleaner movement selection

“keep the vibe”
→ preserve current block identity, weekly rhythm, and broad training feel unless user also requests specific changes

“smoked today”
→ lower fatigue ceiling
→ lower neural demand
→ prefer easier completion
→ maybe trigger lighter-day logic instead of a full rewrite

This translation layer must produce structured programming directives, not just chat wording.

LAYER 4 — CONFLICT / AMBIGUITY HANDLING
Add stronger detection for:
- vague requests
- contradictory requests
- underspecified change requests
- low-confidence interpretation

Examples:
- “make it better” → ambiguous
- “more volume, less fatigue, shorter sessions” → conflicting tradeoffs
- “same vibe but different everything” → contradiction

The system should:
- detect ambiguity
- detect contradiction
- assign confidence
- choose the safest grounded interpretation
- log what tradeoff it made

IMPORTANT:
Do not make the agent freeze on ambiguity.
Do not over-ask for clarification.
Make the best grounded interpretation and audit it.
