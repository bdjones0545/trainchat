FIX 2 — Standardize client mutation invalidation

Create a shared client helper:

handleTrainingSystemMutationResult(result, queryClient, focusMode)

It should invalidate/refetch when:
- result.systemSaved === true
- result.systemEdit?.applied === true
- result.trainingSystemId exists
- result.systemId exists

Invalidate:
- ["training-system-active", focusMode]
- ["training-system-week"]
- ["training-system-history", "changes"]
- ["training-system-history", "versions"]
- ["live-panel-week-ids"]

Use this helper in BOTH:
- SSE complete handler
- non-SSE / JSON response handler

Acceptance:
- Streaming edits refresh panel.
- Non-streaming edits refresh panel.
- Auto-created systems hydrate without reload.

FIX 4 — Verify propagated swaps across weeks

After exercise swap propagation:
- Run verifyMutation() for each propagated week/session change.
- Aggregate verification result.

Store in change log decisionMetadata:

{
  confirmed: boolean,
  propagated: true,
  propagationStatus: "full" | "partial" | "none",
  propagationCheckedCount: number,
  propagationConfirmedCount: number,
  propagationFailedCount: number
}

Rules:
- confirmed = true only if primary swap and all propagated swaps pass.
- propagationStatus = "full" when all propagated swaps verify.
- propagationStatus = "partial" when some fail.
- propagationStatus = "none" when no propagation occurred.

Frontend:
- If partial, do not show a full confirmed badge.
- Show wording like:
  “Swap applied, but some future weeks may need review.”

Acceptance:
- Week 1 swap verified.
- Future week propagated swaps verified.
- Partial propagation is visible, not hidden.
