PART 3 — FIX HIGH-SEVERITY GAPS
==================================================

After auditing, fix the meaningful gaps.

Priority order:
1. Critical
2. High
3. Medium
4. Low

DO NOT waste time on cosmetic-only fixes if system/lifecycle gaps still exist.

Examples of likely high-priority areas:
- session lifecycle persistence
- log session affecting next workout
- week auto-advancement
- next 4-week continuation block generation
- command routing failures
- orphaned chat/program deletes
- hidden or dead-end UX

==================================================
PART 4 — SESSION LIFECYCLE AUDIT
==================================================

Specifically verify and fix:

- Start Session
- Resume Session
- Log Session
- Session Completed state

This must be a real persisted state machine, not a button-label swap.

Expected states:
- not_started
- in_progress
- completed

Expected transitions:
Start Session → Resume Session → Log Session → Session Completed

Make sure it survives:
- leaving and re-entering the screen
- switching tabs
- app reload
- DB fetch

==================================================
PART 5 — LOG SESSION ADAPTATION AUDIT
==================================================

When a session is logged, verify whether the system actually adapts:

Inputs:
- completed / partial / skipped
- session duration
- difficulty
- pain/discomfort
- post-session feeling
- enjoyment
- notes

Audit and fix whether those affect:
- next workout
- rest of week
- current block bias
- agent memory
- chat acknowledgment

Expected:
- too hard / tired → reduce next-session fatigue
- pain → protect movement family and modify future sessions
- too easy / energized → maintain or slightly progress
- skipped / partial → hold or soften progression

==================================================
PART 6 — WEEK ADVANCEMENT AUDIT
==================================================

When all sessions in a week are completed:

Verify and fix:
- week marked complete
- active week advances automatically
- Today / This Week views update
- week role updates (Establish / Build / Intensify / Deload)
- optional lightweight acknowledgment appears

Expected:
W1 complete → W2 active
W2 complete → W3 active
W3 complete → W4 active

No dead-end states.

==================================================
PART 7 — BLOCK COMPLETION + CONTINUATION AUDIT
==================================================

When all 4 weeks are completed:

Audit current behavior and fix gaps.

Expected target behavior:
- mark block complete
- summarize performance of the block
- carry forward memory + constraints + recent performance
- generate the next 4-week continuation block automatically OR prompt for confirmation if needed
- update progression chain
- keep the training story continuous

The next block should not be random.
It should inherit:
- goal
- sport
- age/population
- equipment
- pain history
- adherence
- difficulty trend
- active emphases / biases
- recent exercise exposure
- prior block archetype

==================================================
PART 8 — AGENT COMMAND RELIABILITY AUDIT
==================================================

Audit every major entry point into the agent:

- quick commands
- free text command bar
- sidebar actions
- exercise edit buttons
- block refocus actions
- log session effects
- check-in flows

Verify they all converge into the same real pipeline and do not bypass the system.

Fix any cases where:
- command falls back incorrectly
- “no specific edit identified” appears for supported commands
- quick actions rely on NLP when they should use structured intent
- UI action does not create a visible system result
