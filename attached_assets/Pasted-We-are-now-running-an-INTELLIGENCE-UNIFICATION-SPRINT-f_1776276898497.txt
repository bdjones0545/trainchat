TASK 1 — UNIFY THE EDIT PIPELINE
==================================================

Current problem:
The /system page has a parallel AI edit mechanism that bypasses the main agent pipeline in ai.ts.

This is unacceptable.
All user-initiated training edits — regardless of whether they begin in:
- chat input
- edit drawer
- quick edit action
- global edit action
must flow through the SAME core intelligence pipeline.

I want you to audit:
- system.tsx edit actions
- submitQuickEdit
- submitGlobalEdit
- /api/training-system/edit
- /api/conversations/:id/messages/stream
- ai.ts orchestration

Then refactor so that:
- /system edits are routed through the same intelligent pipeline as /chat
- intent classification, specialist engines, validators, memory, and logging are shared
- there is ONE edit intelligence path in the product

If a legacy direct-edit endpoint must remain temporarily, wrap it so it uses the same backend orchestration layer as the main agent.

Goal:
No more “smart agent in chat, separate edit tool in system.”

==================================================
TASK 2 — INJECT PAGE / UI CONTEXT INTO THE AGENT
==================================================

Current problem:
The agent has no awareness of where the user is in the app or what they are looking at.

I want every relevant message/edit request to optionally carry UI context such as:

type UIContext = {
  page?: "/chat" | "/system" | "/billing" | string;
  selectedWeek?: number | null;
  selectedSessionId?: string | null;
  selectedSessionName?: string | null;
  selectedExerciseId?: string | null;
  selectedExerciseName?: string | null;
  activeProgramId?: string | null;
  activeProgramName?: string | null;
  panelState?: string | null;
};

This context should be:
- sent from frontend where relevant
- included in message/edit payloads
- ingested by the backend orchestration layer
- used in prompt/context building in ai.ts

Examples that must become intelligent:
- on /system, looking at Session 3, user says “make this harder”
- selected exercise is RDL, user says “swap this”
- on /billing, user should be recognized as being in conversion context

The agent should no longer be blind to spatial/product context.

==================================================
TASK 3 — FIX ACTIVE PROGRAM STATE DRIFT
==================================================

Current problem:
Both /chat and /system fetch and manage the active system independently, and /chat also keeps latestProgram in local component state.

This creates drift risk.

I want one canonical frontend source of truth for the active program.

Requirements:
- use the React Query cache for ["training-system-active"] as canonical
- stop keeping shadow copies of the active program in local component state unless absolutely necessary
- if SSE completion returns updated program data, write it into the shared query cache
- /chat and /system must always render from the same canonical state
- invalidation/update strategy must be shared and explicit

Audit:
- fetchActiveSystem in chat.tsx
- fetchActiveSystem in system.tsx
- latestProgram state in chat.tsx
- SSE completion handlers
- cache invalidation logic
- change signal logic

Goal:
The user should never see one version of the program in Coach and another in Your System.
