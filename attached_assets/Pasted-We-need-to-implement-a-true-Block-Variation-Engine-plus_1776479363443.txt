We need to implement a true Block Variation Engine plus a Block Rules Audit system for TrainChat. Right now the generator is over-converging and producing programs that feel too similar even when the exercise details change. The fix is NOT randomization. The fix is structured variation at the block level with auditability.

GOAL
Create a system where program generation first selects a block archetype and split architecture, then builds the program from those rules. Every generated program should feel intentionally different when appropriate, while still respecting user goal, training age, equipment, schedule, recovery, sport intent, and safety constraints.

PRIMARY OBJECTIVE
Stop the app from giving me the same program every time.

HIGH-LEVEL REQUIREMENTS

1. Introduce a Block Variation Engine
The generator must choose from multiple block archetypes before slot filling and exercise ranking happen.

2. Introduce a Block Rules Audit
Every generation should emit a DEV-only structured audit log showing:
- what block archetypes were considered
- why the chosen block won
- what split architecture was chosen
- what rules fired
- what rules were suppressed
- whether fallback logic was used
- whether the resulting program is too similar to recent outputs

3. Preserve coaching quality
Variation must remain constrained and intentional. No random nonsense. The system must still feel like expert programming, not shuffled templates.

BLOCK SELECTION LOGIC

Refactor generation flow so it works in this order:

STEP 1
Read user constraints:
- goal
- strength vs speed mode
- training days
- experience level
- recovery readiness if available
- equipment
- sport tags if present
- session duration
- injury / pain / limitation flags
- current or recent program history if available

STEP 2
Score all BlockArchetypes based on fit.
Example factors:
- goal fit
- schedule fit
- recovery fit
- sport fit
- equipment fit
- novelty fit
- recent repetition penalty
- safety penalty
- mode fit (strength vs speed)

STEP 3
Choose top-scoring valid block archetype.
Do not allow same archetype to win repeatedly unless it is clearly the best fit and alternatives are below threshold.

STEP 4
Within chosen archetype, score valid SplitArchitectures.

STEP 5
Select split architecture with same logic:
- fit
- novelty
- no recent overuse
- rule compliance

STEP 6
Build day templates from the chosen split architecture.

STEP 7
Apply slot rules modified by block archetype.

STEP 8
Rank exercises within each slot using existing candidate metadata plus new block-aware weighting.

STEP 9
Run similarity check against recent programs.

STEP 10
If similarity exceeds threshold, try second-best valid archetype or split before finalizing.

NOVELTY / ANTI-REPETITION RULES

Add anti-repeat logic at multiple levels.

A. Recent block repetition penalty
If recent programs used the same block archetype, apply a penalty.

B. Recent split repetition penalty
If recent programs used the same split architecture, apply a penalty.

C. Recent exercise family repetition penalty
Penalize repeated main lift patterns and repeated assistance families.

D. Recent exact exercise repetition penalty
Penalty should be strongest for exact repeats.

E. Anchor cap
Even default anchor lifts should not dominate unless user profile specifically demands them.

F. Similarity score
Compute a similarity score between newly generated program and last N programs.
Use components like:
- same archetype
- same split
- same top lifts
- same slot ordering
- same movement family emphasis
- same weekly rhythm

If similarity score is above threshold, trigger re-roll logic using next-best valid configuration.

SUCCESS CRITERIA

The implementation is successful if:
1. Generating multiple programs for similar users no longer produces near-identical outputs
2. The selected block archetype materially changes the feel of the program
3. The selected split architecture materially changes weekly rhythm
4. Audit logs clearly explain why a block and split were chosen
5. Similarity detection prevents stale repetition
6. No regressions break current working generation behavior
7. The app still feels like a coach made the program, not a randomizer

IMPORTANT CONSTRAINTS

- Do not break the current working slot system
- Do not remove existing metadata scoring; extend it
- Do not add fake variation that harms program quality
- Prefer clear, typed interfaces
- Centralize all block selection logic
- Use single-source-of-truth principles
- Make all DEV audits easy to grep in console
- If there are any components still bypassing derived program state, identify and fix them
