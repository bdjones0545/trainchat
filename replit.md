# TrainChat — AI Performance Architect

## Overview

TrainChat is an agent-first AI training platform focused on delivering personalized, adaptive training programs and proactive insights through an AI chat interface. Its core purpose is to make expert fitness coaching widely accessible and adaptable to individual user needs. The platform includes a live training workspace for real-time interaction with programs, aiming to revolutionize personalized fitness by providing expert guidance.

## User Preferences

- The product should feel like "vibe coding your training" — a live training workspace, not a standard chatbot.
- 3-panel layout: left sidebar (conversations + nav), center chat, right Live Program Panel.
- The right panel is a tabbed Live Program Panel with: Program, Adapted, Changes, History, and Forecast tabs.
- Empty state shows suggestion chips (pill buttons): "Build my program", "Adjust my split", "Swap exercises", "Reduce fatigue", "Add speed work", "5-day program".
- Program panel header shows a pulsing indicator when a program is active/live.
- When a user has an active training system, the input placeholder changes to "Ask me to adjust your program…" and a green "System active" indicator shows.
- Users get 5 free chat messages before seeing a paywall/signup prompt.
- Mobile: left slide panel for sidebar, right slide panel for Live Program (labeled "Live Program" in header).

## System Architecture

The project is a monorepo built with `pnpm workspaces`, Node.js 24, and TypeScript 5.9. The backend uses Express 5, and the frontend is developed with React, Vite, and Tailwind v4. Data is stored in PostgreSQL with Drizzle ORM, and Zod is used for data validation. API clients are automatically generated from an OpenAPI specification using Orval. The AI core leverages OpenAI's GPT-4o, supported by intelligent fallback mechanisms.

### UI/UX Decisions
The user interface features a dark theme with electric blue accents and the Inter font, designed around a 3-panel chat layout. Key UI elements include chat output, insights displays, and interactive components.

### Technical Implementations
- **AI Core**: An AI Service integrates a Training Intelligence Engine, Adaptation Service, Memory Service, and Insights Service, with a rule-based fallback system.
- **User Management**: Includes plan gating, a guest session system with message limits, and Stripe for subscription billing.
- **Training System**: A persistent, structured training program model with a normalized database schema and dedicated API routes.
- **Natural Language Processing**: An engine interprets natural language to generate structured `EditPlan` JSON for program adjustments.
- **Versioning**: A `system_change_log` tracks all operations for historical viewing and state restoration.
- **Intelligence Layers**: Incorporates Daily Readiness Check-Ins, Post-Session Feedback, a Coach Insights Panel, Collaborative Decision Layer, and a Memory-Driven Collaboration Layer.
- **Program Sync**: Converts AI-generated chat programs into the structured training system.
- **Program Library**: Manages active, saved, and builder sessions.
- **Fast Iteration**: Features a "VibeBar" for quick commands and inline quick actions.
- **Onboarding**: A guest-first onboarding flow allows unauthenticated users to interact with the agent, merging data upon registration.
- **Agent Decision Architecture**: Employs Intent Classification, a Decision Tree for `ActionType` resolution, and a 3-Tier Assumption Confidence system.
- **Exercise Library**: A categorized movement system with specialized fields, a Swap Cluster System, and Progression Links.
- **Knowledge Base**: Stores coaching philosophy, exercise intelligence, and system rules in a `coaching_knowledge` database table.
- **Shareable Program Card**: A "Share" button in the Live Program Panel triggers an AI-generated social card (title, subtitle, phases, day preview, tagline, caption). The backend at `POST /api/share-moments/program-card` uses the program card system prompt with GPT-4o to produce structured JSON; the `ProgramShareCard` component renders a screenshot-ready card with focus-mode theming; `ProgramShareModal` handles loading, preview, save-image, copy-caption, and native share.
- **Gamification (Neural Growth Layer)**: Tracks user progress via `neural_profiles` with XP, levels, and scores.
- **Neural Graph Intelligence Layer**: Interprets `graphState` from `neural_profiles` to generate `NeuralInterpretation` for AI guidance.
- **Program Specialist Decision Layer**: An internal coaching agent converts natural language into structured program adjustments.
- **Conversation Memory**: Extends memory with new types (sport context, time constraint, preferences) for context-aware AI.
- **Prediction Layer**: A real-time pattern recognition engine generates `PredictionSignal`s (e.g., FATIGUE_RISK).
- **Auto-Progression Engine**: Computes `READY_TO_PROGRESS / HOLD / REGRESS` states based on `exercise_logs`.
- **Conditioning / Energy System Engine**: Provides structured energy system programming and session templates.
- **Power & Speed Engine**: Offers intelligence for power and speed programming, including force-velocity zones and session templates.
- **Re-Entry / Return-to-Training Engine**: Detects detrained users and prescribes phased re-entry plans.
- **Block Periodization Engine**: Defines training levels, block types, and goal-specific structures for long-horizon coaching.
- **Hierarchical Program Generation System**: A top-down, 5-layer planning architecture:
  1. **Monthly Block Planner**: Determines block type and outputs `MonthlyBlockPlan`.
  2. **Weekly Block Planner**: Generates 4 `WeeklyBlockPlans` within the monthly context.
  3. **Session Architecture**: Sessions inherit month goal, week emphasis, and session role.
  4. **Exercise Selection**: `selectSlotExercises()` uses `BlockSelectionContext` to enrich intent scoring.
  5. **Reconciliation**: `enforceVariationMandateOnProgram()` as the final enforcement layer.
- **Block Variation Engine**: A 5-module anti-repetition engine preventing repetitive archetypes and exercises, integrated into `buildArchitectureBrief()`.
- **Block Metadata Persistence**: `BlockMetadata` is stored in `training_systems.metadata` JSONB after program generation.
- **Block Phase Hierarchy UI**: Exposes monthly/weekly block context via a breadcrumb line, a compact phase line in the Program tab, and subtle notice bars on specific weeks.
- **Sport-Specific Architecture Engine**: Defines `SportProfile`s for various sports, including physical qualities and season modulation.
- **Return-From-Injury Engine**: Handles users returning from injury with conservative, region-aware programming.
- **Exercise Variation Engine**: Refactored to a family-rotation + exposure-tracking pipeline with a four-tier decision hierarchy for exercise selection, replacing previous penalty-treadmill architecture.
- **Strength Week Expression Engine** (`strength-week-expression.ts`): Auditable metadata-first variant selection for 4-week strength blocks. Per-week configurable thresholds (`SWAP_THRESHOLD_BY_WEEK`: W1=2, W2=999, W3=1, W4=2). Every call returns a `SelectionReason` code (9 codes: library wins from cluster/family, threshold misses, tied candidates, W2 policy, missing metadata). Family fallback now fires when cluster has candidates but none meets the threshold — not only when the cluster is empty. Exports `auditExerciseVariantSelection` (per-candidate breakdowns with rejection reasons), `buildMetadataCoverageReport` (library coverage gaps per primary lift), and `buildFallbackReductionReport` (library vs curated-map ratio by week). Same-family-before-cross-family rule enforced architecturally: library searches only cluster then family; cross-family downgrades exclusively via the curated map. Metadata changes: Back Squat and Conventional Deadlift lowered to stabilityDemand:moderate (base expression); Pause Back Squat, Pause Bench Press, Deficit Deadlift, Paused Deadlift, Pendlay Row, Z-Press raised to stabilityDemand:high + elevated complexity to create score gaps at W3 threshold=1. Barbell Row lowered to stabilityDemand:low so Pendlay Row wins W3. Deficit Deadlift and Paused Deadlift added to deadlift-pattern cluster; Pause Bench Press added to horizontal-push cluster. Verified: W1 100% library-driven, W3 89% library-driven for primary lifts, W4 curated-map-first by policy. Trunk exercises have equivalenceCluster assignments (anti-extension-trunk, anti-rotation-trunk, loaded-carry, lateral-trunk).
- **Strength Session Depth Expander** (`focus-engines/strength-session-depth.ts`): Post-generation hard guarantee that every strength session meets minimum exercise depth. Mirrors the speed-engine depth expander pattern. Session type classifier maps day names to 5 categories (lower, upper, full_body, power_neural, deload). Minimum depth table: working sessions min 5 target 6, deload sessions min 4 target 4. Slot fill order: prep → power → secondary → unilateral → trunk → finisher (deload skips power). Per-category `STRENGTH_AUGMENTATION_BANKS` with concrete, classified exercises for each slot. Called in `ai.ts` within the `focusMode === "strength"` branch of the `isBuildIntent` post-processing pipeline, immediately after the mobility expander. Full `StrengthSessionDepthAudit` trail (session name, type, before/after counts, exercises added, slots filled). `[StrengthSessionDepthAudit]` dev console log scores all sessions on 5 structural dimensions (prepWork, primaryCompound, secondaryWork, unilateral, trunk). Deload floor in `applyStrengthWeekExpression` raised from 2 → 4 so the 60%-slice never collapses a session below usable depth; same floor fixed in `applyWeekProgressionToExercises` for non-strength modes.
- **Harder/Easier OpenAI Fallback Resolver**: A dedicated structured OpenAI resolver for exercise substitutions when local library lacks progressions/regressions. AI-resolved harder/easier relationships are staged in learning candidates and global learning events with `[RelationshipFallbackAudit]`; they no longer mutate canonical `exercise_library.easier_variations` or `exercise_library.harder_variations` unless later promoted through explicit admin/deterministic review tooling.
- **Safe Swap Backstop Layer**: Open-ended exercise swaps remain library-first (cluster → movement pattern), then use approved adjacent movement-family fallbacks, and only then a structured AI proposal. AI-generated swap definitions are validated against movement/equipment/injury constraints, stored as staged learning candidates, and attached to the session exercise metadata without being added to the canonical `exercise_library`.
- **Intent Family System + Transformation Engine**: Structures natural language into 29 `IntentFamily` types with `TargetScope` resolution, mapping to `TransformationBundle`s. Latest additions: `readiness_low` (today-only fatigue), `missed_sessions_reentry` (post-break deload), `environment_temporary_switch` (hotel/home/travel — temporary, not permanent constraint), `sport_context_update` (sport declaration/update), `exercise_dislike_or_preference` (dislike/avoid/prefer signals with permanent storage). Each has FAMILY_PATTERNS, TRANSFORMATION_BUNDLES, and bridgeToSpecialistIntent entries. Anti-routing guard: `hotel gym` removed from `equipment_constraint` patterns so it routes to `environment_temporary_switch` correctly.
- **Adjustment Intent Classifier** (`adjustment-intent-classifier.ts`): Richer classification layer on top of the intent family engine. Produces `AdjustmentIntentClassification` with extracted entities (exercise, equipment, body region, sport, break duration, environment, readiness signal, preference direction), `PersistenceType` (permanent/temporary/session_scope/program_scope/context_update/none), `MutationType` (substitute/remove/add/reduce/increase/deload/adapt_env/store_context/none), `SafetyFlag[]` (pain_adjacent/injury_risk/deconditioning/temporary_only/preference_stored/sport_bias_required/no_rebuild/clarify_before_act), and clarification requirements.
- **Adjustment Execution Planner** (`planAdjustmentExecution()` in `execution-planner.ts`): Richer entry point calling `classifyAdjustmentIntent` → scope resolution → full `AdjustmentExecutionPlan` with action, target, mutationPlan, constraintsToPersist, uiUpdateRequired, verificationRequired, responseMode, fallbackPlan, promptDirective. Routes low-confidence to ASK_CLARIFICATION, guidance families to GUIDANCE/REBUILD_PROGRAM, all mutation families to APPLY_MUTATION.
- **Constraint Compliance Verifier** (`verifyConstraintCompliance()` in `mutation-verifier.ts`): Second-layer semantic verification after mutation. Checks: (1) banned equipment absent from post-mutation snapshot, (2) banned movement patterns absent, (3) pain-region exercises flagged, (4) required exercises present. Returns `ConstraintComplianceResult` with violations list and compliance boolean. Never queries DB — purely synchronous snapshot comparison.
- **Adjustment Response Templates** (`adjustment-response-templates.ts`): Per-family, per-verification-status response strings. Never generic ("change applied"). Templates for all 29 intent families with token substitution ({exerciseName}, {sport}, {environment}, {breakDuration}, etc.). Exports `getAdjustmentResponse()`, `getFailureResponse()`, `getClarificationPrompt()`, `getIntentFamilyLabel()`.
- **Session Identity Sync**: Post-mutation guard ensuring session `label` and `emphasis` consistency.
- **Hierarchical Refinement System**: Extends the edit pipeline with scope-aware mutation routing (`session_scope`, `week_scope`, `block_scope`).
- **Refinement Impact Engine**: Adds intelligent downstream propagation and coach-like acknowledgment to every refinement, classifying impact scopes (`local_only`, `same_week`, `downstream_weeks`, `full_block`).
- **Vibe Coding Interface (Right Panel Behavioral Layer)**: Integrates refinement surfaces (RefinementChips, SessionRefineActions, ProgramRefineInput) into the right panel's Program tab.
- **Focus-Aware Quick Commands**: Right-panel and edit-drawer quick commands are generated through a shared focus/block/session-aware command generator (`getQuickCommands`). Commands are hard-filtered by active focus mode before rendering and emit `[QuickCommandAudit]` logs with focus mode, block type, session intent, and returned commands to prevent strength/speed/mobility cross-contamination. A focus-scoped personalization layer stores command usage separately per user/guest and focus mode, ranks commands by block/session/context relevance, in-focus preference history, in-focus recent history, safety signals, and clutter suppression, and emits `[QuickCommandRankingAudit]` plus `[QuickCommandPreferenceAudit]`.
- **Sports Intelligence System v2**: Expanded sport registry with new sports and role-based subprofiles, using qualitative AI prompt-injection profiles and quantitative `SportDemandProfile` scoring registry.
- **Support Form System**: Three-mode in-app support modal (Contact, Bug Report, Feature Request) with Postgres persistence and SendGrid email notifications.
- **Agent Settings Integration Layer**: User preferences (tone, proactive insights, auto-adjust, memory) are stored in localStorage and injected into the AI system prompt.
- **Privacy & Terms Pages**: Public `/privacy` and `/terms` routes with policy content.
- **Predictive Adaptation + Memory Dominance**: New features for block projection, next session adjustments based on session logs, and memory dominance for AI overrides.
- **Focus Mode Architecture**: Supports isolated training focus lanes (Strength, Speed, Mobility) with shared core functionalities but isolated programming logic, exercise libraries, and AI prompt contexts. Each mode has its own engine in `artifacts/api-server/src/lib/focus-engines/`. The Strength Architecture Brief (`buildArchitectureBrief`) is gated to `focusMode === "strength"` only so Mobility/Speed modes are never contaminated with strength-specific slot architecture.
- **Mobility Engine**: A complete biomotor logic lane for range of motion restoration, positional control, joint prep, movement quality, end-range control, tissue tolerance, recovery flow, and re-entry support. Implemented in `focus-engines/mobility-engine.ts` with 7 block archetypes, 11 movement families, 8 quick commands, and full monthly block planner integration (`MobilityBlockType` system with 10 variants). The exercise library has 60+ dedicated mobility exercises including Hip/Shoulder/Ankle PAILs/RAILs, CARs, Jefferson Curl, breathing drills, dynamic flows, and recovery protocols.
- **System Adjustments Layer** (`lib/system-adjustment-service.ts`, `routes/system-adjustments.ts`, `SystemAdjustmentsPanel.tsx`): Visible adaptation intelligence surface. Persists `SystemAdjustmentEvent` rows in `system_adjustment_events` (PostgreSQL table) whenever meaningful system changes occur — check-in adaptations, block refocuses, fatigue reductions, joint protections. Events are focus-mode-aware (sorted: high-priority → current focus → off-focus → newest). Rendered in the "Adapted" tab of the Live Program Panel with coach-tone titles, one-line summaries, focus-colored icons, relative timestamps, and "New" badges for unseen events. Auto-marks events seen after 4 seconds. Role distinction: Adapted tab = what the system changed; Changes tab = full edit history with snapshots; Coach Insight = why this decision was made right now; Memory = ongoing user truths. Hooked into `check-in-adaptation.ts` (fires after any non-TRAIN_AS_PLANNED check-in adaptation) and `training-system-edit.ts` (fires for structurally meaningful intents: deload_week, travel_mode, adjust_for_pain, refocus_block_*, increase_intensity, increase_weekly_volume, athletic_emphasis, change_frequency, structural_rebuild). Audit logging via `[SystemAdjustmentAudit]` with fields: focusMode, eventType, source, visiblePriority, persisted, displayed. API: GET `/api/system-adjustments?focus=strength&limit=20`, POST `/api/system-adjustments/seen`.
- **Coach Reasoning Layer** (`lib/coach-reasoning-engine.ts`): Deterministic, zero-cost template engine that generates a short 1-2 sentence "Coach Insight" after every build, edit, and check-in adaptation. No additional AI call — pure rule-based template selection keyed by `focusMode × actionType × intent`. Output is injected as `coachReasoning` / `_coachReasoning` into the structured response payload for builds (`_buildMeta._coachReasoning`), edits (`coachReasoning` in `system_edit` data), and check-in adaptations (`AdaptationResult.coachReasoning`). Rendered via `CoachReasoningCallout.tsx` inside `BuildSummaryCard`, `SystemUpdateCard`, and `AdaptationSummaryCard`. Audit logs fire as `[CoachReasoningAudit]` on every generation. Tone is intentionally coach-like: direct, system-aware, no motivational filler. Varies by focus mode (strength = force/structural, speed = mechanics/output, mobility = range/control).
- **Mobility Intelligence Layer (12-Part Parity)**: Full parity with Strength and Speed engines via `focus-engines/mobility-intelligence.ts`. Implements: (1) `MobilityExposureTracker` class for movement family × exercise × week tracking with penalty rules; (2) 16 `MOBILITY_CLUSTER_DEFINITIONS` (hip rotation, hip flexor, hip articulation, hip PAILs, thoracic rotation, thoracic extension, thoracic CARs, shoulder posterior, shoulder anterior, shoulder CARs, ankle passive, ankle active, ankle calf, breathing reset, dynamic flow, recovery) with suppression rules; (3) `MOBILITY_DOSE_PROFILES` (passive_hold, active_control, cars_articulation, breathing_drill, dynamic_flow, recovery_hold) + `MOBILITY_SESSION_CAPS_BY_WEEK`; (4) `RANGE_PROGRESSION_MODEL` — 4-week arc with range depth, hold durations, complexity, control:range ratio, PAILs/loading gates; (5) `JOINT_DISTRIBUTION_TARGETS` — hip 30%, shoulder 20%, spine 20%, ankle 15%; (6) `buildMobilityPredictiveAdaptationContext` — stiffness/pain/fatigue/adherence signal response rules; (7) `buildMobilitySessionLogInterpretationRules` — maps session feedback to programming decisions; (8) `FLOW_PHASE_SEQUENCE` + `buildFlowContinuityDirective` — enforces tissue_prep → activation_CARs → passive_holds → active_control → dynamic_flow → recovery_exit ordering; (9) `MOBILITY_AGENT_LANGUAGE` — preferred/prohibited vocabulary and coach voice examples; (10) Enhanced `MOBILITY_CONTINUATION_RULES` — 12-entry block transition ladder with escalation/de-escalation rules; (11) `buildMobilityParityCheck()` — runtime validation object; (12) `[MobilityParityCheck]` startup log with `"parity":"FULL_PARITY"` on server boot. Monthly block planner enhanced with week-by-week progression rows and continuation block intelligence per block type.
- **Fail-Safe Rules Layer** (`lib/fail-safe.ts`): Shared guardrail classifier for real-world edge-case inputs across chat, streaming chat, direct program edits, and training-system direction generation. Classifies 12 categories including long-horizon requests, excessive frequency, conflicting constraints, vague requests, ambiguous focus, rapid chained edits, partial program state, state mismatch risk, and no-visible-change risk. Applies deterministic strategies (`reduce_scope`, `cap_frequency`, `simplify`, `redirect_focus`, `require_existing_program`, `queue_edit`) before generation or mutation, injects fail-safe prompt context into AI calls, attaches audit metadata to structured responses, prepends coach-facing explanations where appropriate, and uses an in-memory edit lock to prevent overlapping program mutations.
- **Final Response Alignment Verifier** (`lib/response-alignment-verifier.ts`): Pure synchronous post-generation verifier that checks consistency between stage narrations, the action contract, mutation outcome, final program payload, and assistant message text across 6 check types: `guidance_program_leak`, `success_claim_without_save`, `mutation_claim_without_outcome`, `constraint_days_mismatch`, `narration_outcome_mismatch`, `exercise_claim_mismatch`, and `persisted_constraint_violation`. Wired into the SSE path before the final `done` event. On mismatch: attempts transparent repair; on repair failure: returns honest failure description.
- **Cross-Turn Constraint Memory Enforcement** (`lib/constraint-memory.ts`): Automatically persists user-stated constraints to long-term memory and enforces them in all future program generations. Four exported functions: `persistConstraintsFromTurn` (fire-and-forget on every turn, maps constraint-family intent signals → `upsertMemory` calls for `equipment_constraint`, `exercise_dislike_or_preference`, `sport_context_update`, `injury_modification`, `joint_friendly_modification`); `loadHardConstraints` (extracts `{bannedItems, dislikedItems, painRegions, sport}` from already-loaded MemoryEntry list with no extra DB call); `buildConstraintEnforcementDirective` (formats hard constraints as an absolute AI prompt enforcement section, injected at the front of `transformHint` in both SSE and non-SSE paths); `validateAgainstHardConstraints` (post-generation checker, returns violations for banned/disliked items found in program exercises). Wired into conversations.ts for ALL plan tiers (Pro users: uses already-loaded memories; non-Pro: lightweight separate query). Temporary constraints (e.g. hotel gym, persistenceType="temporary") are explicitly skipped. The alignment verifier is extended with a `persisted_constraint_violation` critical check type and corresponding repair message.
- **Duplicate-Safe Add Exercise Layer** (`lib/edit-engine.ts`): The single database writer for `add_exercise` now blocks same-day duplicate and near-duplicate exercise additions across all producers. It uses the approved exercise library first, scoring alternatives by movement pattern, adjacent movement family, focus mode, and week/block reuse penalties. If no safe library option exists, a constrained AI backstop may propose a validated staged candidate; generated candidates are written only to learning/audit tables and never promoted into the canonical exercise library. Audit logs emit `[AddExerciseAudit]` and `[DuplicateExerciseBlockAudit]`.
- **Constraint-Aware Exercise Selection Engine** (`lib/exercise-constraint-filter.ts`): Moves constraint enforcement upstream into the exercise selection/scoring process so invalid exercises are structurally impossible to select — not caught and repaired post-generation. Three pure, synchronous functions: `conflictsWithPainPattern(exerciseName, painRegions)` maps pain regions (knee/shoulder/lower back/hip/wrist/elbow/ankle) to movement-pattern keyword groups with hard (contraindicated) and soft (caution) severity levels; `filterCandidatesByConstraints<T>(pool, hardConstraints)` removes banned items and disliked items BEFORE the softmax scoring draw with a non-empty safety fallback guarantee; `computeConstraintPenalties(exerciseName, hardConstraints)` computes penalty scores for the scoring layer (banned=100, disliked=6, pain hard=6, pain soft=3). Threaded through the full call chain: `hardConstraints` added to `ScoreContext`, `CandidateScore.breakdown`, and `ExerciseScoreBreakdown.factors`; pre-filtering wired into `pick()` inside `selectSlotExercises()`; `selectSlotExercises` extended with `hardConstraints?` param; `buildArchitectureBrief` extended with `hardConstraints?` param (all 4 weekly calls); `AIResponseOptions.hardConstraints?` added and destructured in `generateAIResponse`; wired into all 4 `generateAIResponse` call sites in `conversations.ts` (non-SSE primary + retry, SSE primary + retry) with `hardConstraintsNonSSE` / `hardConstraintsSSE` already loaded by the cross-turn constraint memory layer. 45 new tests, 878 total passing.

### System Design Choices
- **Database Schema**: Comprehensive tables for user data, training programs, conversation history, analytics, and coaching knowledge.
- **Authentication**: Session-based using `express-session` with a PostgreSQL store, supporting anonymous users via `deviceId`.
- **Anonymous User Architecture**: All visitors are immediately assigned an anonymous user account, with data merging upon registration.
- **Anonymous → Paid Upgrade Flow**: An `AnonymousUpgradeModal` facilitates inline account creation and data preservation during checkout.
- **Training System Auto-Creation**: Automated persistence of AI-generated programs after `CREATE_PROGRAM` or `START_NEW_PROGRAM` intents.
- **Intelligence Unification**: A unified context pipeline (`UIContext`) ensures consistent memory synchronization for all AI calls.
- **DEFAULT EXECUTION LAYER**: Resolves natural-language references to concrete database entity IDs before AI processing.
- **Modularity**: Services are designed as distinct modules with clear extension points.
